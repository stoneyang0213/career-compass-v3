# 职业测评报告生成 —— 严谨性规范

本规范定义本产品如何在规模化、无人工逐份审核的前提下,保证每一份自动生成的报告达到科学严谨的下限。它配套一套代码骨架(report_generator / fact_checker / pipeline),代码是骨,本规范是脑。

## 一、核心原则:架构约束,而非指令请求

生成式模型的本质是生成"看起来合理"的文本,它没有"我不确定这个数字所以不写"的内在机制。因此严谨性**不能靠在 prompt 里请求模型严谨来实现**——再长的指令也挡不住它在不知道的地方流畅地编。

我们的做法是从数据流上约束:把"事实"和"解读"彻底分开。模型只允许在解读层自由发挥;在事实层,它只能引用我们维护的结构化数据库,**无权生成任何数字或外部事实**。

## 二、事实与解读的边界

- **事实层(模型不得创造)**:用户测评分数(MBTI、霍兰德、价值观等)= 用户的一手事实,可信;行业数据 = 只能来自 `industry_facts.json`,带出处、口径、适用范围。
- **解读层(模型可自由发挥)**:基于上述事实的职业推理、画像描述、行动建议。

任何统计数字、百分比、外部机构结论,只要不在事实库里,模型就不许写。

## 三、行业事实库:字段含义与准入标准

每条数据必须填全字段(见 `data/industry_facts.json`)。重点字段:

- `original_scope`:原始口径与适用范围。这是防"放大"的核心。例:WEF 教育增长数据必须注明"主要由低收入国家人口驱动",防止被用来论证中国一线成人市场。
- `applies_to`:`macro_direction`(宏观方向)或 `specific_market`(具体市场)。宏观数据严禁用于论证个人细分市场的付费需求。
- `confidence`:high/medium/low,**按真实核实情况如实填**。不确定就填 low,不要为了让报告显得严谨而虚标。
- `usage_note`:这条数据的使用禁区。高风险数据必须写明。
- `verified_at` / `expires_at`:核实日期与复核截止日。过期数据系统自动剔除并告警,复核后才能恢复使用。

**准入标准**:新增条目必须能追到一手出处(原始报告 + 发布日期 + 链接),追不到的不准入库。二手转述(博客、社媒、Medium)不能作为 source,只能作为发现线索。

## 四、为什么校验是独立的一步

报告生成后,跑一次职责单一的校验调用(`fact_checker.py`):换一个"校对员"身份重审。即使是同一个模型也有效,因为它此时不再背"把报告写流畅"的压力,只盯事实对不对。校验提取报告里每个数字,逐一比对库内条目、口径、宏观/微观误用。

## 五、对用户的诚实(模板层固定文案)

- 每条职业推荐旁:本路径真实可行性需由你自己的种子用户付费数据验证。
- 报告首/尾:本报告为 AI 基于你的测评与公开趋势生成的推理性建议,非行业研究报告,数据仅供方向参考;重大决策请结合线下专业咨询与从业者访谈。

这不是免责,而是严谨性本身的一部分:把局限说在前面,提升专业可信度。

## 六、责任分工(必须落到具体的人)

| 事项 | 责任方 | 频率 |
|---|---|---|
| 维护事实库、新增条目核实 | 内容/产品团队 | 持续 |
| 数据到期复核 | 内容/产品团队 | 按 expires_at,行业数据约每半年 |
| prompt 模板变更评审 | 产品 + 工程 | 每次变更 |
| 校验结果人工抽查 | 内容团队 | 上线初期每份,跑稳后抽样 |

> 关键提醒:事实库的维护是产品方的持续成本,不能甩给模型。对一个帮人做重大职业决策的产品,这是它能不能立住的底线——用户拿报告去赌的是自己的人生路径,一个编出来的数字可能误导一次真实的转行决定。

## 七、落地优先级

1. **改造一 + 二(地基,先做)**:结构化事实库 + 带约束生成。完成即可消除绝大部分"编数据"问题。
2. **改造三(高性价比补强)**:独立校验。上线初期用人工审核模式,跑稳后再自动化。
3. **改造四(零成本,立即上线)**:模板层固定文案 + 三版块结构。

## 八、可选的进阶项

若技术栈允许给校验环节接入联网搜索,可实时验证出处真伪,而不仅比对自有库——但增加成本与延迟。建议初期不接,先用"比对自有库"的最小可行版本跑顺。
